💡
この資料の目的：上記の懸念に対して「Entra IDでどこまで実現できるか」「どこが限界か」を整理し、次回定例で説明するための根拠資料として使用する。

結論（先にまとめ）

  • ✅ 階層構造（ネスト型グループ）：完全に実現可能。すでに提案済みの設計で対応できる
  • ✅ 兼務設定：機能的には実現可能。ユーザーを複数グループに所属させることで同等の効果が得られる
  • ⚠️ ガルーンとの視覚的な差異：「兼務」という明示的なラベルは存在しない。組織図的な表示はManagerフィールドの一本鎖に限られる
  • 💡 代替手段あり：カスタム属性（extensionAttribute）で兼務を記録し、動的グループで自動管理する方式が有効

🏛️ 結論：階層構造は完全に実現可能

✅
Entra ID セキュリティグループはネスト（入れ子）構造をサポートしています。すでに提案済みの設計がそのまま適用できます。

Entra IDでは、グループの中に別のグループをメンバーとして追加できます（最大ネスト深度：制限なし）。これにより「大学全体 → 大部門 → 小部門」という階層を表現できます。

🌳 提案済みのネスト型グループ構造

提示済みの5層構造案（再掲）：

MAU_全教職員 ← 最上位：全員対象の一斉連絡・SharePointアクセス用
├── MAU_職員_全体
│ ├── MAU_職員_事務局_教務部
│ ├── MAU_職員_事務局_総務部
│ ├── MAU_職員_事務局_学生部
│ └── MAU_職員_市ヶ谷キャンパス
└── MAU_教員_全体
├── MAU_教員_造形学部_油絵学科
├── MAU_教員_造形学部_工業デザイン学科
├── MAU_教員_造形構想学部
└── MAU_教員_非常勤講師

この構造の動作メカニズム

  • 「MAU_全教職員」に何かアクセス権を付与すると、配下の全グループのメンバー全員に自動的に権限が継承される
  • 「MAU_職員_事務局_教務部」だけに権限を付与すると、教務部のメンバーだけがアクセスできる
  • ユーザーは下位グループ（末端）に追加するだけで、上位グループの権限も自動的に得られる

📌 SharePoint への適用例

SharePointサイト アクセス付与対象グループ 結果
大学ポータル（全体） MAU_全教職員 全教職員・非常勤含む全員
教務部専用サイト MAU_職員_事務局_教務部 教務部員のみ
造形学部教員向けサイト MAU_教員_造形学部_油絵学科 + MAU_教員_造形学部_工業デザイン学科 + … 対象学科教員のみ
💡
SharePoint の「対象ユーザー設定」（オーディエンス ターゲティング）でも Entra ID グループを指定できるため、ナビゲーションや告知記事の表示を部門別に切り替えることも可能です。

⚙️ 技術的な制約（参考）

  • ネスト上限：Microsoft 365 グループはネスト不可だが、セキュリティグループはネスト可能（今回はセキュリティグループで設計）
  • SharePoint のオーディエンスターゲティングはセキュリティグループを直接参照可能
  • Teams の通知・チャネルは独立したグループ管理だが、セキュリティグループをメンバー追加の基準にできる
  • 動的メンバーシップルール（Entra ID P1 = A3ライセンス以上で利用可能）を使えば、ユーザー属性変更時にグループ所属が自動更新される

🔄 結論：兼務設定は機能的に実現可能

✅
Entra IDではユーザーは複数のグループに同時所属できます。これにより「造形学部の教授が大学院研究科も兼任」「職員が複数部署を兼務」といったケースに対応できます。
⚠️
ガルーンのような「兼務」という明示的なラベル表示はありません。あくまでグループへの所属で権限・アクセスを管理します。

📖 兼務の仕組み（概念説明）

ガルーンでは「主所属」と「兼務所属」を明示的に区別します。Entra IDではすべて「グループメンバー」として同等に扱います。

📋
6/18定例で出た実際の例（井上さん）：
「監察チーム（チームリーダー）」かつ「総務グループ → 施設管理チーム → 情報管理担当（チームリーダー）」を兼務している。
ガルーンでは 総務グループ/井上、施設管理チーム/井上、情報管理担当/井上、監察チーム/井上 と、それぞれ別エントリとして全て登録されている。
この「縦軸（職位・雇用形態）× 横軸（部署・チーム）」の二次元構造をM365で実現できるか、が核心的な懸念点。

具体例：教務部を兼任する工業デザイン学科教授

👤 田中 太郎 教授
→ 所属グループ：
MAU_教員_造形学部_工業デザイン学科
MAU_職員_事務局_教務部

結果として、田中教授は：

  • 工業デザイン学科向けの SharePoint サイトにアクセスできる
  • 教務部向けの SharePoint サイトにもアクセスできる
  • MAU_全教職員 の一括連絡を受け取れる
  • 教務部の Teams チャネルにも参加できる

別例：研究科と学部を兼任する教員

👤 鈴木 花子 教授
→ 所属グループ：
MAU_教員_造形構想学部
MAU_教員_造形学部_油絵学科

⚙️ 実装方法：3つのアプローチ

  1. A
    手動追加（推奨・シンプル） 管理者がユーザーを複数のセキュリティグループに手動で追加する。グループ設計がシンプルならこれで十分。
    ✅
    追加ライセンス不要。即実施可能。
  2. B
    動的メンバーシップルール（自動化） ユーザーの属性（department、jobTitle、employeeType等）に基づいてグループ所属を自動化。 
    // 例：工業デザイン学科の教員を自動追加するルール
user.department -eq "造形学部_工業デザイン学科"
-and user.userType -eq "Member"

// 例：兼務グループの自動化（extensionAttribute を使う場合）
user.extensionAttribute1 -contains "教務部兼務"
    ⚠️
    Entra ID P1 ライセンスが必要（A3/A5、または個別購入 約680円/月/ユーザー）
  3. C
    カスタム属性での兼務記録（推奨・現実的） ユーザーオブジェクトの extensionAttribute（拡張属性）に兼務情報を記録し、管理台帳として活用する。
    💡
    extensionAttribute は15個あり、自由に活用できます。例：
    extensionAttribute1 = "工業デザイン学科"（主所属）
    extensionAttribute2 = "教務部兼務"（兼務）
    これを台帳として管理し、グループ追加の根拠にする。

🔍 ガルーンとの「兼務」概念の違い

井上さんの懸念は「ガルーンの兼務が再現できるか」でした。以下が正確な答えです：

観点 ガルーン Entra ID（M365）
主所属の概念 明示的に設定可能 なし（全グループ同等）
兼務の概念 明示的に「兼務」として登録 なし（複数グループ所属で代替）
権限・アクセス効果 主+兼務の両方が付与される 複数グループの権限が全て付与される
組織図表示 兼務を視覚的に表示可能 Managerフィールドの1系統のみ（兼務表示は非対応）
メーリングリスト 各所属グループから受信 各グループから受信（同等）
管理の煩雑さ ガルーンの管理画面で一元管理（現在は畠中氏が人事異動のたびに手作業で更新） Entra ID管理センターで個別追加が必要。動的グループ（P1）で自動化すれば現状より改善できる
💡
井上さんへの説明ポイント：
「ガルーンの兼務が持っている機能的な効果（権限・メール受信・アクセス）はEntra IDでも全て実現できます。
一方で視覚的な組織図への兼務表示はM365では標準機能では対応していません。組織図は主所属（Managerフィールド）の1系統のみの表示になります。
この点をどこまで重視するかを確認してください。」

⚠️ ガルーン移行で注意すべき差異

  1. 1
    組織図の見え方が変わる
    ガルーンでは兼務が視覚的に見えていたが、M365の組織図（TeamsのPeople/Delve）はManagerフィールドの1系統のみ表示。兼務は見えない。
    → 組織図を「人事の台帳」として使っていたなら代替手段が必要
  2. 2
    グループ管理の手間が増える可能性
    ガルーンでは人事情報との連動で自動更新されていた場合、M365では都度手動でグループ追加・削除が必要になる。
    → 動的グループ（Entra ID P1）か、人事システム連携（Azure AD Connect等）で対応可能
  3. 3
    「主所属」の概念の扱い
    Entra IDにはどのグループが「主所属」かを示す標準フィールドがない。
    → departmentフィールドを主所属として使い、extensionAttributeで補足情報を管理するのが現実的

⚙️ 推奨設計案：Phase別アプローチ

💡
現時点では動的グループ（P1要件）は使わず、まず手動管理からスタートし、運用に慣れてから自動化を検討する段階的アプローチを推奨します。

Phase 1 — 基本グループ設計（即実施可能）

  1. 1
    グループ命名規則の確定
    MAU_[種別]_[大分類]_[小分類] の形式で統一
    MAU_全教職員
    ├── MAU_職員_全体
    │ ├── MAU_職員_事務局_教務部
    │ ├── MAU_職員_事務局_総務部
    │ └── MAU_職員_市ヶ谷
    ├── MAU_教員_全体
    │ ├── MAU_教員_造形学部_[各学科]
    │ ├── MAU_教員_造形構想学部_[各学科]
    │ └── MAU_教員_非常勤講師
    └── MAU_MFA対象_教職員 ← セキュリティポリシー用
  2. 2
    兼務ユーザーの管理台帳作成
    Excelまたは SharePoint リストで「兼務者一覧」を管理。列：氏名・UPN・主所属グループ・兼務グループ1・兼務グループ2
  3. 3
    テストアカウントで動作確認
    井上さん（畠中幹太さん経由）から提供予定のテストアカウントを使い、複数グループ所属とSharePointアクセス確認

Phase 2 — 運用改善（8月以降・状況に応じて）

  1. A
    extensionAttribute の活用
    ユーザーの主所属・兼務情報をEntra IDの拡張属性に記録し、管理台帳代わりにする 
    // Entra ID ユーザープロパティの例
department: "工業デザイン学科"           // 主所属
extensionAttribute1: "教員"               // 種別
extensionAttribute2: "教務部兼務"          // 兼務情報

  2. B
    動的グループの検討（A3ライセンス前提）
    人事情報の定期的な変更に対応するため、departmentフィールド基準の動的グループを導入。人事異動時のグループ更新を自動化。
    ⚠️
    現在のライセンス体系（A1）では利用不可。A3への拡張か個別のEntra ID P1追加購入が必要（約680円/月/ユーザー）
