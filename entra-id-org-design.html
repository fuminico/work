🔄 兼務の実装方法（多重グループ所属）

Garoonで「複数組織に同時所属」しているユーザーは、Entra IDでは 複数のグループに同時登録 することで同等の機能を実現します。

職員が 総務G_経理チーム の主所属 ＋ 100周年事業推進室 を兼務 MAU_事務_総務G_経理チーム ＋ MAU_事務_100周年事業推進室 に同時登録 本来業務と兼務先の両方のリソースにアクセスできる

主所属の管理方法

⚠️
Entra IDには「主所属」「兼務」の区別がありません。userオブジェクトの department フィールドを主所属として使い、兼務はグループ多重登録で管理するのが現実的です。

ユーザー属性の使い方：
・department: 主所属組織（例: 総務グループ_経理チーム）
・jobTitle: 役職（教授/准教授/専任職員/嘱託職員 等）
・employeeType: カテゴリー（教員/職員[専任]/助手 等）
・officeLocation: 拠点（鷹の台/市ヶ谷）
・extensionAttribute1: 兼務先1（任意で記録）

⚠️ 設計上の留意点

対象留意事項
理事長・業務執行理事・常務理事 Garoonでは「組織」として登録されているが、実態は個人の役職。Entra IDでは jobTitle フィールドに記載し、グループは作成しない（少数の個人）。
学生相談室・保健室（カテゴリー） 組織と役割の両方の性質を持つ。事務組織の下に配置するか、カテゴリーとして扱うか確認が必要。
100周年事業推進室 時限プロジェクト組織。グループ作成時に Entra ID グループの説明欄に廃止予定日を記載し、定例レビューで削除する。
教学組織はフラット Garoon上「造形学部」「造形構想学部」という上位分類はない（各学科が直接並んでいる）。Entra IDグループも同様のフラット設計でよい。学部軸が必要になれば後から追加する。
通信教育・派遣職員 通常の専任教職員とライセンス要件が異なる可能性がある。カテゴリーグループ分けによりライセンスを別管理する。
事務組織のチーム構成（確認中） 学生支援グループ等のチーム構成は後日確認・追記予定。判明次第このページに反映する。

🏛️ 結論：Garoon の3階層構造はEntra IDで完全に実現可能

✅
Entra ID セキュリティグループはネスト（入れ子）構造をサポートしています。Garoonの「グループ → チーム → 担当」の3階層をそのままグループの入れ子で表現できます。

ユーザーは末端グループ（チームまたは担当）にのみ登録します。上位グループへのネストにより、権限・メールは自動的に継承されます。

🔄 結論：兼務設定は機能的に実現可能

✅
Entra IDではユーザーは複数のグループに同時所属できます。これにより「工芸工業デザインの教員が情報教育センターも兼担」「職員が複数チームを兼務」といったケースに対応できます。
⚠️
ガルーンのような「兼務」という明示的なラベル表示はありません。あくまでグループへの所属で権限・アクセスを管理します。

📖 兼務の仕組み（概念説明）

ガルーンでは「主所属」と「兼務所属」を明示的に区別します。Entra IDではすべて「グループメンバー」として同等に扱います。

📋
6/18定例で出た実際の例（井上さん）：
「監査チーム（チームリーダー）」かつ「総務グループ → 施設管財チーム → 情報管理担当（チームリーダー）」を兼務している。
ガルーンでは 総務グループ/井上、施設管財チーム/井上、情報管理担当/井上、監査チーム/井上 と、それぞれ別エントリとして全て登録されている。
この「縦軸（職位・雇用形態）× 横軸（部署・チーム）」の二次元構造をM365で実現できるか、が核心的な懸念点。

⚙️ 実装方法：3つのアプローチ

  1. A
    手動追加（推奨・シンプル） 管理者がユーザーを複数のセキュリティグループに手動で追加する。グループ設計がシンプルならこれで十分。
    ✅
    追加ライセンス不要。即実施可能。
  2. B
    動的メンバーシップルール（自動化） ユーザーの属性（department、jobTitle、employeeType等）に基づいてグループ所属を自動化。 
    // 例：工芸工業デザイン学科の教員を自動追加するルール
user.department -eq "工芸工業デザイン"
-and user.userType -eq "Member"

// 例：兼務グループの自動化（extensionAttribute を使う場合）
user.extensionAttribute1 -contains "教学企画チーム兼務"
    ⚠️
    Entra ID P1 ライセンスが必要（A3/A5、または個別購入 約680円/月/ユーザー）
  3. C
    カスタム属性での兼務記録（推奨・現実的） ユーザーオブジェクトの extensionAttribute（拡張属性）に兼務情報を記録し、管理台帳として活用する。
    💡
    extensionAttribute は15個あり、自由に活用できます。例：
    extensionAttribute1 = "工芸工業デザイン"（主所属学科）
    extensionAttribute2 = "教学企画チーム兼務"（兼務）
    これを台帳として管理し、グループ追加の根拠にする。

🔍 ガルーンとの「兼務」概念の違い

井上さんの懸念は「ガルーンの兼務が再現できるか」でした。以下が正確な答えです：

観点 ガルーン Entra ID（M365）
主所属の概念 明示的に設定可能 なし（全グループ同等）
兼務の概念 明示的に「兼務」として登録 なし（複数グループ所属で代替）
権限・アクセス効果 主+兼務の両方が付与される 複数グループの権限が全て付与される
組織図表示 兼務を視覚的に表示可能 Managerフィールドの1系統のみ（兼務表示は非対応）
メーリングリスト 各所属グループから受信 各グループから受信（同等）
管理の煩雑さ ガルーンの管理画面で一元管理（現在は畠中氏が人事異動のたびに手作業で更新） Entra ID管理センターで個別追加が必要。動的グループ（P1）で自動化すれば現状より改善できる
💡
井上さんへの説明ポイント：
「ガルーンの兼務が持っている機能的な効果（権限・メール受信・アクセス）はEntra IDでも全て実現できます。
一方で視覚的な組織図への兼務表示はM365では標準機能では対応していません。組織図は主所属（Managerフィールド）の1系統のみの表示になります。
この点をどこまで重視するかを確認してください。」

⚠️ ガルーン移行で注意すべき差異

  1. 1
    組織図の見え方が変わる
    ガルーンでは兼務が視覚的に見えていたが、M365の組織図（TeamsのPeople/Delve）はManagerフィールドの1系統のみ表示。兼務は見えない。
    → 組織図を「人事の台帳」として使っていたなら代替手段が必要
  2. 2
    グループ管理の手間が増える可能性
    ガルーンでは人事情報との連動で自動更新されていた場合、M365では都度手動でグループ追加・削除が必要になる。
    → 動的グループ（Entra ID P1）か、人事システム連携（Azure AD Connect等）で対応可能
  3. 3
    「主所属」の概念の扱い
    Entra IDにはどのグループが「主所属」かを示す標準フィールドがない。
    → departmentフィールドを主所属として使い、extensionAttributeで補足情報を管理するのが現実的
